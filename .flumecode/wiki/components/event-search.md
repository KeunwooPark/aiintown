# AI event search

> **TL;DR** — A scheduled GitHub Actions job runs a Python script that asks Claude (with web search) for upcoming offline AI events per city and commits the results as per-city JSON under `_data/events/`.

> **Purpose** — Periodically discover upcoming in-person AI events for a configurable list of cities and store them as committed data files.
> **Key files** — `_data/cities.yml`, `scripts/search_events.py`, `scripts/dedupe_events.py`, `scripts/requirements.txt`, `.github/workflows/search-ai-events.yml`, `scripts/test_search_events.py`, `logs/search_runs.jsonl`.
> **Depends on** — the Anthropic Messages API + its server-side `web_search` tool (`ANTHROPIC_API_KEY` secret); GitHub Actions. **Used by** — the Jekyll site, which renders the JSON on the homepage (see [components/site.md](components/site.md)).
> **Related** — [architecture.md](architecture.md) for how this sits beside the Pages build, [glossary.md](glossary.md) for terms.

## The one core idea

This subsystem **generates data, not pages**. A daily job collects events with an LLM and commits them into the repo as `_data/events/<city-id>.json`. Because the files live in Jekyll's `_data/` directory, the site renders them at build time via `_layouts/home.html` (see [components/site.md](components/site.md)) — this subsystem stays focused on producing clean data. The search is deliberately scoped to *upcoming* events; finished events are never written.

## The parts

### `_data/cities.yml` — the input

The single human-editable source of truth for which cities to search. It is a YAML list; each entry has `id` (slug used as the output filename), `name`, `country`, `enabled`, an optional `language` (a human-readable name like `Korean` or `German`, no locale codes), an optional `locale` (a short locale **code** like `ko` or `de`), optional `query_hints` (keywords injected into the prompt), and an optional `event_platforms` list of local/regional event-listing platforms for that city. Adding or removing a city is a pure data edit — no code change — and the script iterates every entry where `enabled` is not `false`.

The two language-ish fields have **distinct jobs**: `language` (human-readable name) drives the search prompt; `locale` (code, matching the keys in `_data/i18n.yml`) is consumed only by the Jekyll layer — templates stamp it on each city section and event card so client-side JS can show a city's localized event text when the active UI locale matches (see [components/site.md](components/site.md)). `query_hints` mixes English topic/format terms with local-language general keywords (e.g. Seoul carries `밋업`/`세미나`, Berlin carries `Treffen`/`Vortrag`) so coverage broadens rather than shifts. The hints stay topic-/format-focused; language selection lives in the dedicated `language` field, not in `query_hints`.

For **every** city, `search_city` names the built-in global platforms (`GLOBAL_EVENT_PLATFORMS = ["Luma", "Eventbrite", "Meetup"]`, a module-level constant) in the prompt as additional sources to consult — *in addition to, not instead of,* general web search. A city may also declare an optional `event_platforms` list of local/regional providers, which is appended to the global list for that city's prompt (e.g. Seoul uses `["Onoffmix", "Festa", "EventUs"]`, Berlin uses `["Eventbrite Berlin", "Meetup Berlin"]`). The appended sentence reads: *"In addition to general web search, also check dedicated event-listing platforms such as {platforms} for matching events."* A city with no `event_platforms` field still gets a valid prompt naming only the three global platforms. Note that `web_search_20250305` accepts no site allow-list, so this steering biases — but does not guarantee — which sites Claude visits.

### `scripts/search_events.py` — the worker

For each enabled city the script builds a prompt, calls Claude, parses the response, filters, and merges into the city's JSON file. The important design points:

- **Web search is mandatory.** The request passes the `web_search_20250305` server-side tool (name `web_search`, `max_uses: 8`). Without it Claude cannot browse and would hallucinate events. `MODEL` is pinned in one constant (`claude-sonnet-4-5`).
- **Output ceiling sized to avoid truncation.** The call sets `max_tokens=8000`. This matters because a JSON array truncated mid-element makes `parse_json_array` return `[]` and silently drop the *whole* batch — the ceiling is deliberately roomy so large per-city responses are not lost. Raising `max_uses` to `8` likewise lets the agent run more searches per city and reach less-obvious sources. Both increase Anthropic credit usage per run.
- **Run-date-scoped prompt.** `RUN_DATE` (today) is interpolated into the prompt, which instructs Claude to return *only* upcoming, in-person AI events on or after that date as a JSON array of `title`, `date`, `venue`, `url`, `source`, `description`, plus the localized `title_local` and `description_local` (see the next bullet).
- **Localized, community-aware prompt.** When a city defines `language`, `search_city` appends a sentence telling Claude to also search the web and read sources in that language (in addition to English) to catch locally-indexed events; the same `if language:` branch also instructs that `title_local`/`description_local` be the title and description written in `{language}` (copied verbatim if the event is already in that language). Both the local-language search sentence and the `*_local` instruction are skipped when `language` is absent, keeping the prompt back-compatible. A further sentence — added for every city — tells Claude to include small, grassroots, and community events (university clubs, coworking spaces, local meetup/community sites and venues) *in addition to, not instead of,* larger conferences and international events. A final sentence steers the search toward dedicated event-listing platforms (the global `GLOBAL_EVENT_PLATFORMS` plus any per-city `event_platforms`; see the `_data/cities.yml` section above). The `web_search` config and `parse_json_array`/`merge` handling are unchanged — `parse_json_array` keeps every key the model returns and `merge` spreads it through, so the new `*_local` fields flow into the written JSON with no whitelist change, and non-ASCII titles still round-trip via the `ensure_ascii=False` write path.
- **Single-event-per-URL steering.** For *every* city `search_city` appends an instruction telling Claude to return each distinct event only **ONCE** — never the same event under different title phrasings or from multiple sources — and to use the event's primary/official page as `url`, merging duplicate sources into one item with a canonical URL. This attacks the duplicate problem at the source so fewer same-event/different-title records ever reach `merge`.
- **"Find new events" steering.** To stop the model re-surfacing the same well-known events, `_existing_event_entries(city)` reads the city's already-stored *upcoming* events as `(title, url)` pairs (capped at `EXISTING_TITLE_LIMIT = 50`, soonest first since records are stored sorted by `(date, title)`) and — only when the list is non-empty — `search_city` appends them to the prompt formatted as `title — url`, instructing the model to skip events already on file *by URL as well as title* and prioritize ADDITIONAL events. First-run cities and the no-`id` unit-test cities get no such sentence, so the prompt stays back-compatible. The helper returns `[]` defensively on a missing `id`, missing file, or malformed JSON. Note this biases but does not guarantee novelty — `web_search_20250305` accepts no allow/deny list.
- **Lazy client.** The `Anthropic()` client is created inside `main()`, not at import, so the test suite can import the module without `ANTHROPIC_API_KEY`.
- **Defensive parsing.** `parse_json_array` extracts the array from possibly-prose output — preferring a fenced ```` ```json ```` block, else the first `[` … last `]` span — and returns `[]` (never raises) on malformed output, dropping any item lacking a `title`. This is what keeps a bad LLM response from corrupting the data files.
- **Code-level upcoming guard.** `is_upcoming` drops events whose parseable `date` is strictly before `RUN_DATE`, but *keeps* events with missing/unparseable dates (they could still be upcoming). `_apply_date_status` tags those kept-but-undated events with `date_status: "unknown"` so consumers can surface the uncertainty.
- **URL-based dedupe + history merge.** The dedup key is the **normalized URL within a city**, not the title — this is the core of the duplicate fix. `event_id` is a 16-char SHA-1 of `url|<normalized-url>|<city>` when the record has a usable URL, falling back to the legacy `title.strip().lower()|date|city` key only when there is no URL. So two records pointing at the same event page collapse into one even when their titles differ (e.g. `ICML 2026` vs `International Conference on Machine Learning (ICML 2026)`); URL-less records keep their old per-title behavior. `merge` re-indexes the existing file *through `event_id`* (so duplicates already on disk collapse on load), then upserts each upcoming `found` event by the same key. On a collision it calls `_combine` to pick the survivor and forces `last_seen = RUN_DATE_STR` while keeping the earliest `first_seen`; on a new id both timestamps are set to `RUN_DATE_STR`. Records are written sorted by `(date, title)` with their recomputed `id` re-stamped (the site and downstream data key on `id`, so every written record must carry it). Because `merge(city, [])` re-indexes and collapses with no new input, it doubles as the one-time cleanup primitive (see `scripts/dedupe_events.py` below).
- **`normalize_url` — conservative canonicalization.** A standard-library helper (`urllib.parse`, no new deps) that returns `""` for falsy/non-string/empty input and otherwise: prepends `https://` when no scheme is present, lowercases the host and strips a leading `www.`, **drops the scheme** so `http`/`https` compare equal, drops the fragment, strips known tracking params (`TRACKING_PARAMS`: `utm_*`, `fbclid`, `gclid`, `mc_cid`, `mc_eid`, `ref`, `ref_src`) and sorts the rest, and strips a trailing `/` from the path. It is deliberately conservative — it only collapses forms clearly pointing at the same resource and preserves remaining query params.
- **`_combine` — choosing the survivor.** When two records map to the same `event_id`, `_combine` returns the canonical one: prefer the record with a parseable date (`date_status != "unknown"` and non-empty `date`), then the longer `description`, with a deterministic `title` tiebreak. The survivor always inherits the **earliest `first_seen`** and **latest `last_seen`** across both, so the dedup never narrows an event's seen-window. The discarded variant's title/description wording is lost — no alias field is kept (a deliberate trade-off).
- **Per-run observability counts.** `merge` does double duty as a counter: while iterating `found` it tallies `fetched` (`= len(found)`), `skipped_past` (items failing `is_upcoming`), `duplicates` (upcoming items whose `event_id` already existed in `by_id`), and `new` (upcoming items added for the first time), and **returns** that counts dict keyed by `city`. All prior merge behavior is unchanged — the counting is purely additive. `main()` collects one counts dict per enabled city into a `per_city` list and passes it to `write_run_log`. This exists to answer the diagnostic question "does re-running actually add events, or just re-find the same ones?" — high `duplicates` with `new == 0` run after run points at a coverage gap, not a dedup bug.

### `logs/search_runs.jsonl` + `write_run_log` — the run log

`write_run_log(per_city)` appends **one JSON object per run** to `logs/search_runs.jsonl` (created on first run via `LOGS_DIR.mkdir`). Each entry is `{"run_date": RUN_DATE_STR, "cities": [...per-city counts...], "totals": {...summed fetched/new/duplicates/skipped_past...}}`. The file deliberately lives at the **repo root under `logs/`, not under `_data/`**, so Jekyll does not load it into `site.data` on every build. Append mode keeps every prior run, making the file a growing, diffable history; an entry is written **even on no-op runs**, so repeated re-triggers that change nothing are still visible. The same function also `print`s a per-city line (`<city>: fetched=… new=… duplicates=… skipped_past=…`) plus a `TOTAL:` line to stdout, so the counts surface directly in the GitHub Actions run log without needing to open the committed file. The run date reuses the existing `RUN_DATE_STR` constant — no new time source. Constants `LOGS_DIR = pathlib.Path("logs")` and `RUN_LOG = LOGS_DIR / "search_runs.jsonl"` sit next to `DATA`/`EVENTS_DIR`.

`scripts/requirements.txt` pins the two runtime deps (`anthropic`, `PyYAML`). `scripts/test_search_events.py` unit-tests the pure helpers (`is_upcoming`, parsing, merge, `normalize_url`) and the prompt-construction path (mocking `client`) without network or API key — including that `normalize_url` collapses `www.`/scheme/trailing-slash/tracking-param/fragment variants, that two records sharing a normalized URL but differing in title get the same `event_id` while URL-less records still split by title, that URL-duplicates collapse both within one `merge` call and across an on-disk re-index (`merge(city, [])`) while preserving the earliest `first_seen`/latest `last_seen`, that the prompt contains the `only ONCE` single-event instruction, plus the pre-existing existing-titles/`*_local` assertions. A `TestMerge` group also asserts the counts dict `merge` returns (all-`new` on a fresh file, all-`duplicates` on a re-run, `skipped_past` for a past-dated item), and a `TestWriteRunLog` group (monkeypatching `LOGS_DIR`/`RUN_LOG`/`RUN_DATE_STR` to a temp dir) checks that `write_run_log` writes one valid JSON line with correct `totals`, appends a second line on a second call, and handles an empty `per_city`.

### `scripts/dedupe_events.py` — the one-time cleanup

A small standalone maintenance command (`python scripts/dedupe_events.py`) that collapses the duplicate records already accumulated in the committed `_data/events/*.json` files. It loads `_data/cities.yml` and, for **every** city (regardless of `enabled`) whose JSON file exists, calls `se.merge(city, [])` — passing an empty `found` list so `merge` re-indexes the existing records through the new `event_id` and collapses URL-duplicates via `_combine` without adding anything. It imports the production functions from `scripts.search_events`, so the cleanup and the daily run can never diverge, and prints a `city: before -> after` count for each file. It introduces no new dependency. The implementer ran it once for this change, which collapsed the known same-URL/different-title groups (e.g. ICML and Festival of Learning in Seoul, CONNECTED Berlin and a Language-AI meetup in Berlin), reducing `seoul.json` from 26 to 13 records and `berlin.json` from 34 to 23. Past-dated records are preserved — only the daily-search `found` list is filtered by `is_upcoming`, never the existing records.

### `.github/workflows/search-ai-events.yml` — the schedule

A GitHub Actions workflow triggered daily by `schedule` (cron `17 3 * * *`) and on demand via `workflow_dispatch`. It checks out the repo, sets up Python 3.12, installs `scripts/requirements.txt`, runs the script with `ANTHROPIC_API_KEY` from `secrets`, then commits any change back to `main` (the commit is skipped when `git diff --cached --quiet` finds no diff). The `Commit results` step stages **both** `_data/events` and `logs` (`git add _data/events logs`) so the appended run log reaches `main` alongside the event data; because a new log line is appended every run, this step normally always has something to commit. The job needs `permissions: contents: write` to push.

### `_data/events/<city-id>.json` — the output

One JSON array per city, generated by the script. Each record's shape is documented in the repo `README.md`. These files are the deliverable, and the site's `_includes/event-card.html` reads exactly these field names (`title`, `date`, `venue`, `url`, `source`, `description`, `date_status`, plus the optional `title_local`/`description_local` localized variants). Existing files generated before this change simply lack the `*_local` keys and fall back to English until the next run rewrites them — no backfill is performed.

## Gotchas and constraints

- **Web search is billed.** Every daily run consumes Anthropic API credits; the tool counts toward usage.
- **Undated noise persists.** Keeping events with vague/missing dates avoids losing real ones but means some non-upcoming clutter (flagged `date_status: "unknown"`) can remain.
- **No retroactive pruning.** Only *new* past events are filtered at write time; events already stored stay as accumulated history even after their date passes.
- **Run log grows unbounded.** `logs/search_runs.jsonl` gains one appended line per run and is never pruned or rotated; over time it becomes a large committed file. A future retention/rotation policy is out of scope for now.
- **Counts diagnose, they don't fix.** The run log reveals *whether* duplicates are the cause of "re-running adds nothing", but broadening actual coverage (e.g. surfacing more Luma/Meetup events) is a separate follow-up.
- **URL key can over-merge.** The dedup key is `url|city`, not `url|date`, so a single page reused across dated instances (e.g. a Meetup group page, or a generic landing/aggregator URL shared by genuinely different events) collapses them into one record. This is the accepted trade-off; the `before -> after` counts printed by `dedupe_events.py` are the safety check for eyeballing an unexpected collapse before committing.
- **Migration changes `id` values.** Records previously stored under the old `title|date|city` id get recomputed (URL-based) ids after the cleanup. The ids are internal-only here, so this is safe, but any external reference to an old `id` would break.
- **Push to `main` needs write access.** If branch protection is enabled on `main`, the commit step fails and the workflow/permissions must be adjusted.
- **Cron is best-effort.** GitHub Actions `schedule` can be delayed under load and is auto-disabled after ~60 days of repo inactivity.

## Extension points

- **Add/remove cities** by editing `_data/cities.yml` only.
- **Change the model or search budget** via the `MODEL` constant, the tool's `max_uses`, and `max_tokens` (the output ceiling that guards against truncation).
- **Tune the "find new events" steering** via `EXISTING_TITLE_LIMIT` (how many already-stored title/url entries get injected into the prompt).
- **Tune URL dedup aggressiveness** via `TRACKING_PARAMS` (which query params are stripped) and the `normalize_url` rules. To keep recurring dated instances separate, the `event_id` key would need to fold `date` back in alongside the URL.
- **Re-run the cleanup** with `python scripts/dedupe_events.py` after any change to the dedup logic to re-collapse the committed data files.
- **Adjust how events render** in the Jekyll layer — `_layouts/home.html` / `_includes/event-card.html` (see [components/site.md](components/site.md)).
